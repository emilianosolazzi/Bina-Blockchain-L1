Unified Entropy Pipeline

▸ click to hide/show All sources active
—

Three independent entropy streams are harvested and mixed into every beacon output. PoW Mining mints TGBT on Arbitrum L2 (via MiningModule → TokenomicsModule). Stale Block rewards are now live — the StaleBlockOracle submits proofs and claims TGBT automatically (base 50 TGBT × quality × fork depth). UTXO anchoring contributes entropy and Bitcoin-grade timestamps.

Why three entropy sources?

Unlike Chainlink VRF (which relies on a single oracle's secret key), this beacon mixes three independent physical entropy sources to make the randomness impossible to predict or manipulate:

  • PoW Mining: CPU-intensive hash search — the nonce solution is inherently unpredictable
  • Dead UTXO Anchoring: Uses spent Bitcoin transaction outputs as entropy seeds — tied to real-world Bitcoin finality
  • Stale Block Mining: Harvests orphaned Bitcoin blocks — fork events that are physically impossible to predict

The three streams are XOR-mixed with multiple rounds, producing randomness that is at least as strong as the strongest individual source.

💀 Dead UTXO Anchoring

Active
Anchors your beacon entropy to Bitcoin's immutable chain — giving every output a timestamp backed by 15+ years of proof-of-work security. The miner queries mempool.space and blockstream.info for four types of permanently unspendable outputs: spent (already consumed), OP_RETURN (data-only, unspendable by design), dust (too small to ever spend), and burn-address (sent to known dead wallets like 1BitcoinEater…). It selects the best candidate by entropy-weighted score, then SHA-256 hashes the UTXO data with OS randomness + Bitcoin block headers (enterprise entropy). The resulting hash is XOR'd into the beacon.
What does anchoring to Bitcoin give you?

By tying each entropy output to a specific dead Bitcoin UTXO at a known block height, you get:

  • Bitcoin-grade timestamps — proves your randomness existed before a specific Bitcoin block. Stronger than Ethereum alone because Bitcoin has deeper finality.
  • Tamper-proof provenance — the anchor hash is derived from an output that can never change (it's dead). Anyone can verify the anchor independently via the public Bitcoin chain.
  • Cross-chain entropy mixing — Bitcoin's unpredictable UTXO patterns add a source of entropy that is physically separate from the Arbitrum L2 chain, making manipulation exponentially harder.
  • Future utility — the same anchoring infrastructure enables document notarisation, IoT hardware attestation, software build provenance, and legal evidence timestamping (see create_entropy_anchor_with_reference() in utxo_fetcher.rs).

This is a plus that other randomness beacons don't have — your entropy is anchored to real Bitcoin finality, not just Ethereum.

The simple version

📜 Solidity — Lottery that picks a winner using Temporal Gradient randomness 
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

interface IBatchMining {
    function verifyRandomnessLeaf(
        uint256 epochId,
        uint256 leafIndex,
        bytes32 outputHash,
        bytes32[] calldata proof
    ) external view returns (bool);
}

contract FairLottery {
    IBatchMining constant BEACON =
        IBatchMining(0xAf07E37D104E9be17639FE7a51B36972D4738651); // Arbitrum

    address[] public players;

    function enter() external payable {
        require(msg.value == 0.01 ether, "Entry is 0.01 ETH");
        players.push(msg.sender);
    }

    /// @notice Anyone calls this with a verified random number to pick the winner.
    ///         The random number comes from the Temporal Gradient miner — not Chainlink.
    function pickWinner(
        uint256 epochId,
        uint256 leafIndex,
        bytes32 outputHash,      // ← the random number from the dashboard above
        bytes32[] calldata proof  // ← the Merkle proof from the Proof Inspector
    ) external {
        require(players.length >= 2, "Need players");

        // ✅ Verify the randomness is real and finalized on-chain
        require(BEACON.verifyRandomnessLeaf(epochId, leafIndex, outputHash, proof),
                "Invalid randomness proof");

        // 🎲 Use the hash to pick a winner (modulo player count)
        uint256 winnerIndex = uint256(outputHash) % players.length;
        address winner = players[winnerIndex];

        // 💰 Pay the winner
        delete players;
        payable(winner).transfer(address(this).balance);
    }
}

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC721/ERC721.sol";

interface IBatchMining {
    function verifyRandomnessLeaf(
        uint256, uint256, bytes32, bytes32[] calldata
    ) external view returns (bool);
}

contract RandomTraitNFT is ERC721 {
    IBatchMining constant BEACON =
        IBatchMining(0xAf07E37D104E9be17639FE7a51B36972D4738651);

    uint256 public nextTokenId;
    mapping(uint256 => uint8) public rarity; // 0=common, 1=rare, 2=legendary

    constructor() ERC721("RandomNFT", "RNFT") {}

    function mint(
        uint256 epochId,
        uint256 leafIndex,
        bytes32 outputHash,
        bytes32[] calldata proof
    ) external {
        // Verify the random number is legit
        require(BEACON.verifyRandomnessLeaf(epochId, leafIndex, outputHash, proof),
                "Bad proof");

        uint256 tokenId = nextTokenId++;
        uint256 roll = uint256(outputHash) % 100;

        // 🎨 Assign rarity based on the random number
        if (roll < 60)      rarity[tokenId] = 0;  // 60% common
        else if (roll < 90) rarity[tokenId] = 1;  // 30% rare
        else                rarity[tokenId] = 2;  // 10% legendary

        _mint(msg.sender, tokenId);
    }
}

🦀 Rust — Verify randomness off-chain 
use alloy::primitives::{B256, U256};
use alloy::providers::ProviderBuilder;
use alloy::sol;

sol! {
    #[sol(rpc)]
    interface IBatchMining {
        function verifyRandomnessLeaf(
            uint256 epochId,
            uint256 leafIndex,
            bytes32 outputHash,
            bytes32[] proof
        ) external view returns (bool valid);
    }
}

#[tokio::main]
async fn main() -> eyre::Result<()> {
    let provider = ProviderBuilder::new()
        .on_http("https://arb1.arbitrum.io/rpc".parse()?);

    let beacon = IBatchMining::new(
        "0xAf07E37D104E9be17639FE7a51B36972D4738651".parse()?,
        provider,
    );

    // These values come from the API or the dashboard
    let epoch_id = U256::from(4);
    let leaf_index = U256::from(9);
    let output_hash: B256 = "0x000d8a2dc892...".parse()?;
    let proof: Vec<B256> = vec!["0xc8d9...".parse()?, "0xab12...".parse()?];

    let valid = beacon.verifyRandomnessLeaf(epoch_id, leaf_index, output_hash, proof)
        .call().await?.valid;

    println!("Randomness verified: {valid}");  // true ✓

    // Use it: convert to a number
    let roll = (U256::from_be_bytes(output_hash.0) % U256::from(100)) + U256::from(1);
    println!("Random 1-100: {roll}");
    Ok(())
}

What are epochs and why do they matter?

An epoch is a batch of miner solutions grouped into a Merkle tree and committed on-chain. Think of it like a "block" of randomness:

  • Solutions accumulate — the miner finds PoW solutions and stores them locally
  • Epoch commitment — once enough solutions (default: 10) are collected, the epoch builder creates a Merkle tree and commits the root hash on-chain via commitEpochRoot()
  • Challenge window — a 28,800 L1-block (~96 hour) waiting period allows anyone to challenge invalid epochs
  • Finalization — after the challenge window passes, finalizeEpoch() is called, locking the epoch permanently and minting TGBT rewards
  • Storage attestation — optionally, the epoch data is attested on-chain to prove the full Merkle tree is stored and retrievable

Benefits: Epochs batch gas costs (one on-chain commit per 10 solutions instead of per-solution), provide a tamper-proof audit trail, and enable consumers to verify any past randomness output via its Merkle proof.

For Chainlink VRF users: Each epoch is like a VRF subscription fulfillment — but you can verify any output from any epoch, forever, without paying per-verification.
